Proposal 1

Research -> single-file HTML deck

What survives scrutiny

  • Forces synthesis instead of raw note dumping.
  • Gives you a portable artifact you can open anywhere.
  • Fits comparisons, recommendations, architecture options, and technical investigations.
  • Single-file HTML is a good constraint: simple to store, inspect, and share.

What needs tightening

  • Beautiful is too vague. Replace it with concrete standards.
  • Not every tiny lookup deserves a slide deck.
  • The deck needs a fixed structure so quality does not depend on mood or model style.

Better standard

  • Self-contained single HTML file.
  • No external dependencies.
  • Readable typography and clear section hierarchy.
  • Explicit recommendation, tradeoffs, risks, and evidence appendix.

Where the rule should apply

  • Non-trivial research
  • Comparisons and vendor evaluations
  • Architecture option analysis
  • Recommendation memos for decisions with real cost
Good default for complex research. Bad default for quick factual lookups.

Proposal 2

Web UI features -> navigable demo artifact

Why the idea is strong

  • It proves discoverability, not just isolated functionality.
  • It catches orphan screens and broken navigation.
  • It gives fast review evidence for UX and workflow quality.
  • It aligns with the rule that E2E flows should navigate through the UI.

Critical constraint

tests prove correctness; demo artifacts prove reachability and usability

A smooth demo can still hide broken validation, auth gaps, state bugs, or unhappy-path failures.

Scope it correctly

  • Apply it to user-facing web UI features.
  • Do not force it on backend-only, plumbing-only, or tiny non-visual changes.
  • For internal tools, use judgment: demo only if the UI itself is part of the deliverable.

Required flow

  • Start from the app main page.
  • Reach the feature through visible navigation.
  • Show the core happy path.
  • Show the visible result or saved state.
  • Use video when possible, annotated screenshots or an HTML walkthrough when not.
This is a review artifact for UI work, not a substitute for automated verification.

Decision

What belongs in AGENTS.md vs skills

Put this in AGENTS.md

  • Which task categories should produce review artifacts.
  • What those artifacts are meant to prove.
  • Fallback expectations when tooling is unavailable.
  • The rule that artifacts supplement tests and written summaries.

Put this in skills

  • How to build the deck well.
  • How to record or assemble the demo artifact well.
  • Section templates, quality bars, navigation requirements, and fallback workflows.
  • Tooling guidance for generating HTML, screenshots, or video files.

Recommended skill split

research-slide-deck

For non-trivial research, comparisons, recommendations, and architecture-option analysis.

webapp-feature-demo

For completed user-facing web features that need a reachability and usability walkthrough.

Policy in one place. Operational detail in reusable workflows. That is the clean split.

Suggested wording

Global rule: concise, explicit, enforceable

For complex deliverables, produce a review artifact suited to the task. Non-trivial research should default to a self-contained single-file HTML slide deck unless the user asks for another format. User-facing web-app features should include a UI walkthrough artifact showing how the feature is reached and used from the main page or existing navigation. Prefer video; if video capture is unavailable, use annotated screenshots or an HTML walkthrough. These artifacts supplement tests and written summaries; they do not replace them.

Why it works

It defines expectations without hard-coding one brittle production process into AGENTS.md.

Why it avoids failure

It gives the model a fallback path instead of pretending video or browser tooling will always be available.

Why it stays honest

It explicitly keeps testing and reasoning as first-class evidence rather than letting presentation masquerade as proof.

Short enough for AGENTS.md. Specific enough to shape behavior.

Extensions

More complex task categories

Backend / API feature

Ship a runnable API demo pack: example requests, responses, auth notes, happy path, and one failure path.

Bug fix

Ship a repro + fix evidence bundle: reproduction, root cause, before/after evidence, tests added, regression risk.

Refactor

Ship an architecture delta brief: what changed, what did not, boundary shifts, and proof behavior stayed intact.

Performance

Ship a benchmark report with baseline, environment, workload, variance, charts, and tradeoffs.

Data migration

Ship a migration readiness pack: assumptions, dry-run results, edge failures, rollback plan, sample before/after rows.

Security-sensitive work

Ship a threat-model memo: attack surface, trust boundaries, abuse paths, mitigations, and residual risk.

Different task classes need different proof artifacts. One format for everything is lazy.

Bottom line

Recommended next move

Adopt now

  • Add one short policy paragraph to global AGENTS.md.
  • Create a research-slide-deck skill.
  • Create a webapp-feature-demo skill.
  • Treat both outputs as review evidence, not proof of correctness.

Do not do

  • Do not require decks for trivial research.
  • Do not require videos for backend-only work.
  • Do not use vague words like beautiful without explicit quality criteria.
  • Do not let polished demos excuse missing tests.

The idea is worth adopting because it improves review quality. It only becomes bad when phrased as a universal rule instead of a task-specific artifact policy with well-defined workflows.

Best implementation: short policy + two dedicated skills + explicit fallbacks.
